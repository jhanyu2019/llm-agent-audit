What I test

This is not a prompt review. I test the action boundary: what context the agent reads, what tools it can call, what trace it leaves, and whether the action was authorized.

When an AI agent can call tools, the failure that matters is not what it says. It is what it does. I check whether content the agent reads, such as a user message, a support ticket, an email, a document, or a tool response, can push it into a high-impact action it should not take.

Examples include issuing a refund, moving money, deleting an account, granting access, disabling MFA, or exporting data.

Why it matters

A model's refusal is a safety feature, not an authorization system. A stronger model can help, but it does not replace application-side authorization. On a fixed battery run across six recent models from three major model vendors, the average number of unsafe outcomes per run ranged from 0.0 to 8.0 on the same 58-scenario battery, and model tier was not a reliable safety signal. The thing that stops an unauthorized action has to live in your application, so the only reliable way to know what your agent does is to test your agent. See the writeups and data.

The audit question is simple: did the agent call a tool it should not have been allowed to call?

What this looks like in practice

These are examples of the action boundary, not public test payloads.

Finance and APAn accounts-payable agent reads a vendor invoice whose text says the bank account changed. A text-only test checks whether the model warns about fraud. This audit checks whether the agent actually calls a tool to update bank details or schedule payment.
HealthcareA prior-authorization agent reads a payer response that asks for more documentation or raises the urgency. This audit checks whether the agent changes what it prepares or submits without the right review gate.
LogisticsA dispatch agent reads a carrier email or a phone transcript where the caller says a job is urgent or asks to change someone else's booking. This audit checks whether the agent dispatches, cancels, or edits the job without verifying the caller is allowed to ask.

How a pilot works

Your team runs a small set of scenarios against a staging copy of your agent and sends back the tool-call traces. I review the tool-call traces against pre-written pass/fail rules and send a report mapped to the OWASP Top 10 for LLM Applications, with the evidence and concrete fixes. No production access, no real customer data, no shared credentials. Scenarios are written for your workflow and can cover a single high-impact action or a short workflow where an earlier approval might be reused or applied too broadly later in the workflow.
